    Introducción Git

    La siguiente información fue tomada de: https://git-scm.com/book/es/v1

    ¿Qué es el control de versiones, y por qué es importarte?, El control de versiones es un sistema que registra los cambios realizados sobre un archivo o conjunto de archivos a lo largo del tiempo, de modo que puedas recuperar versiones específicas más adelante.

    Sistema de control de versiones Locales

    • Un método de control de versiones usado por mucha gente es copiar los archivos a otro directorio (quizás indicando la fecha y hora en que lo hicieron, si son avispados).
    • Es fácil olvidar en qué directorio te encuentras, y guardar accidentalmente en el archivo equivocado o sobrescribir archivos que no querías.
    • Se desarrollaron soluciones usando simples base de datos en la que se llevaba registro de todos los cambios realizados sobre los archivos
    • Con estas herramientas se pueden recrear cómo era un archivo en cualquier momento.

    Sistemas de control de versiones centralizados

    • Este tipo de sistemas surgió bajo la necesidad de colaborar con desarrolladores en otros sistemas, (Centralized Version Control Systems o CVCSs en inglés)
    • Tienen un único servidor que contiene todos los archivos versionados, y varios clientes que descargan los archivos desde ese lugar central.
    • Durante muchos años éste ha sido el estándar para el control de versiones (Siglo XX).
    • (Ventaja) Todo el mundo puede saber (hasta cierto punto) en qué están trabajando los otros colaboradores del proyecto.
    • (Ventaja) Los administradores tienen control detallado de qué puede hacer cada uno; y es mucho más fácil administrar un CVCS que tener que lidiar con bases de datos locales.
    • (Desventaja) La desventaja La más obvia es el punto único de fallo que representa el servidor centralizado.
    • (Desventaja) Cuando tienes toda la historia del proyecto en un único lugar, te arriesgas a perderlo todo.

    Sistemas de control de versiones distribuidos

    • Como solución a los sistemas centralizados se crean (Distributed Version Control Systems o DVCSs en inglés). En un DVCS (como Git, Mercurial, Bazaar o Darcs)
    • Permite establecer varios flujos de trabajo que no son posibles en sistemas centralizados, como pueden ser los modelos jerárquicos.
    • Puedes colaborar con distintos grupos de gente simultáneamente dentro del mismo proyecto.
    • Cada vez que se descarga una instantánea, en realidad se hace una copia de seguridad completa de todos los datos.

    ¿Que es Git?

    Es un software de control de versiones diseñado por Linus Torvalds, pensando en la eficiencia y la confiabilidad del mantenimiento de versiones de aplicaciones cuando éstas tienen un gran número de archivos de código fuente. https://es.wikipedia.org/wiki/Git

    Historia:

    • Durante la mayor parte del mantenimiento del núcleo de Linux (1991-2002), los cambios en el software se pasaron en forma de parches y archivos.
    • En 2002, el proyecto del núcleo de Linux empezó a usar un DVCS propietario llamado BitKeeper.
    • En 2005, la relación entre la comunidad que desarrollaba el núcleo de Linux y la compañía que desarrollaba BitKeeper se vino abajo y arranco el desarrollo de Git como herramienta para versionamiento de códigos.

    Fundamentos de Git

    Para entender un poco mejor como almacena los datos, explicaremos brevemente el modelo de almacenamiento de Git, esta herramienta modela sus datos más como un conjunto de instantáneas de un mini sistema de archivos. Cada vez que confirmas un cambio, o guardas el estado de tu proyecto en Git, él básicamente hace una foto del aspecto de todos tus archivos en ese momento, y guarda una referencia a esa instantánea. Para ser eficiente, si los archivos no se han modificado, Git no almacena el archivo de nuevo, sólo un enlace al archivo anterior idéntico que ya tiene almacenado.

    • Casi cualquier operación es local, la mayoría de las operaciones en Git sólo necesitan archivos y recursos locales para operar. Por lo general no se necesita información de ningún otro ordenador de tu red.
    • Git no necesita salir al servidor para obtener la historia y mostrártela, simplemente la lee directamente de tu base de datos local.
    • Esto también significa que hay muy poco que no puedas hacer si estás desconectado o sin VPN. Si te subes a un avión o a un tren y quieres trabajar un poco, puedes confirmar tus cambios felizmente hasta que consigas una conexión de red para subirlos

    Los tres estados (Repositorio Local)

    Esto es lo más importante a recordar acerca de Git si quieres que el resto de tu proceso de aprendizaje prosiga sin problemas. Tomado de: http://www.slideshare.net/origamiaddict/git-get-ready-to-use-it

    Esto nos lleva a las tres secciones principales de un proyecto de Git: el directorio de Git (Git directory), el directorio de trabajo (working directory), y el área de preparación (staging area).

    Tomado de: http://www.slideshare.net/origamiaddict/git-get-ready-to-use-it

    El flujo de trabajo básico en Git es algo así:

    1. Modificas una serie de archivos en tu directorio de trabajo.
    2. Preparas los archivos, añadiendolos a tu área de preparación.
    3. Confirmas los cambios, lo que toma los archivos tal y como están en el área de preparación, y almacena esas instantáneas de manera permanente en tu directorio de Git.

    Trabajando en red

    • Clone: Clonar un repositorio significa bajarse una copia completa del mismo a nuestro a ordenador, después de clonar un repositorio tendremos en nuestro computador toda la historia del repositorio, completa.
    • Push: Cuando tu proyecto se encuentra en un estado que quieres compartir, tienes que enviarlo a un repositorio remoto. El comando que te permite hacer esto es sencillo hacerlo con un push.
    • Pull: Si has configurado una rama para seguir otra rama remota (véase la siguiente sección y el Capítulo 3 para más información), puedes usar el comando pull para recuperar y unir automáticamente la rama remota con tu rama actual.
    • Fetch: Recupera toda la información enviada a ese servidor desde que lo clonaste (o desde la última vez que ejecutaste fetch). Es importante tener en cuenta que el comando fetch sólo recupera la información y la pone en tu repositorio local —no la une automáticamente.

    Mas información en: https://git-scm.com/book/es/v1/Fundamentos-de-Git-Trabajando-con-repositorios-remotos

    Creación de repositorios

    Un repositorio es la unidad básica de GitHub, más comúnmente es un proyecto. Los repositorios pueden contener carpetas y archivos, incluyendo imágenes o cualquier tipo de información o documento que requiera.

    Para crear un nuevo repositorio

    1. Hacer Clic the icon next to your username, top-right.
    2. Escribir el nombre de su nuevo repositorio.
    3. Escribir una corta descripción
    4. (Recomendable) Seleccionar Initialize this repository with a README, esta opción construye un archivo con la descripción general del repositorio.

    Reporte de Errores

    Un problema (Issue) es una nota en un repositorio de algo que necesita atención. Podría ser un error, una solicitud de funcionalidad, una pregunta, entre otras cosas.

    Abrir un problema

    1. Haga clic en la pestaña de Problemas en la barra lateral.
    2. Dar Clic en Nuevo Problema (New Issue).
    3. Escribir el problema, un titulo y la descripción.

    Creación de Branch

    Cuando se crea una rama, se está realizando una copia independiente de la rama original, como estaba en un determinado punto del tiempo (ver siguiente figura). Si existen cambios en la rama principal mientras se esta trabajando en una nueva rama, no se preocupe, siempre se puede revertir o integrar esos cambios.

    Para crear una nueva rama (Branch).

    1. Abrir el repositorio en donde creara la nueva rama.
    2. Hacer clic en el menú desplegable en la parte superior de la lista de archivos que dice branch : master.
    3. Escriba un nombre de la nueva rama , readme-edits, en el cuadro de texto.
    4. Seleccione la opción crear una rama ... o pulsar "Enter" en su teclado.

    Hacer Commit

    Ejemplo hacer Commit

    1. Hacer clic en el archivo README.
    2. Hacer clic en el icono del lápiz en la esquina superior derecha de la vista de archivos para editar.
    3. En el editor, escribir un texto de su preferencia.
    4. Escribe un mensaje en el formulario que describe los cambios a confirmar.

    Crear Pull Request

    Paso Captura de pantalla
    Dar clic en el icono (Pull Request) ubicado en la barra lateral de la pagina, luego en la pagina de Pull request, hacer clic sobre el botón verde New pull request.
    Seleccionar la rama (brach) que quiere integrar, readme-edits, para comparar con el rama master (la rama original).
    Verificar todos los cambios en la pagina de comparaciones, debe tener la certeza que los cambios corresponden a lo que se quiere subir en al repositorio.
    Si ud esta satisfecho con los cambios, es decir son los cambios que quiere enviar, debe hacer clic en el botón verde Create Pull Request button.
    Se define un titulo, y si este corresponde a un problema o a un error abierto, se debe incluir en el titulo "fixes #" y el número de incidencia. También es aconsejable escribir una breve descripción de los cambios.
